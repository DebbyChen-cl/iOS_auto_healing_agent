AI 自動化測試 Auto-Healing 系統導入計畫

目標產品：PHD iOS 影像編輯 App UI 自動化測試
計畫書 複雜度：High 目標成熟度：L3 產生時間：2026-06-24 15:10

一、主管摘要

本計畫不是要讓 AI 直接取代測試判斷，而是建立一層 AI Test Maintenance Layer，協助自動化測試在失敗時收集證據、分析 root cause、修復低風險測試路徑 drift，並產出可審核的 HTML 報告與 PR。

第一階段以 Auto-Healing / Failure Handling 為主，先解決 UI 自動化測試維護成本高、失敗原因判斷耗時、測試結果容易被 network/server busy 或 dependency cascade 污染的問題。

系統的核心原則是保護品質訊號：AI 可以修復低風險測試路徑 drift，但不修改 assertion、不自動 merge。AI 修復後通過的 case 會標記為 pass_with_healing，保留原始 failure evidence，並產出 HTML 報告供人工審核。

二、要解決的問題與設計想法

目前 PHD iOS App 的 UI 自動化測試有一個核心矛盾：測試越完整，維護成本越高；但如果只是讓 AI 把失敗測試修到通過，又可能掩蓋真正的產品問題。因此本系統要解決的不是「讓測試變綠」，而是縮短從失敗到可信判斷的時間。

目前問題造成影響本系統的處理方式
UI locator、page object、流程小改動容易讓測試失敗工程師需要花時間重現、看截圖、找元素、修改測試碼AI 分析 failure evidence，對低風險 drift 產生 patch，replay 通過後開 PR
server busy / network error 很常發生測試報告混入非產品問題，降低 failure 判讀效率獨立標註 network_issue / server_busy，依 retry policy 處理，不誤判為一般 failure
只有一台 iOS 測試裝置若每個 fail 都即時 healing，整體 AT 時間會大幅拉長Fail 當下即時收 evidence 與分類，需要操作 App 的 healing 進入 device-aware queue
測試之間存在資料或流程依賴A 失敗可能導致 B/C/D 都失敗，報告看起來像大量獨立 fail建立 dependency analysis，downstream case 標為 blocked_by_upstream
AI 修復如果缺乏審核，會有品質風險可能把產品 bug 或測試意圖改變誤修成 pass不自動 merge，所有成功 healing 產出 HTML 報告與 PR，由人工審核後 merge
設計想法：將 AI 放在「測試維護層」，而不是「測試判定層」。測試 pass/fail 仍由 test runner 決定；AI 負責分析、建議、修補低風險測試碼，並提供可審核證據。

三、導入目標

  • 降低 iOS UI 自動化測試維護成本，尤其是 locator、page object、等待條件與微小流程 drift。
  • 保留測試作為品質訊號的可信度，不讓 AI 把真正產品問題修成表面 pass。
  • 建立 failure evidence 與 baseline evidence，供後續 AI 分析與 future test case generation 使用。
  • 用 HTML 報告作為主要審核介面，讓 reviewer 能快速判斷 AI 修復是否可信。

四、不包含範圍

  • 不允許 AI 修改 assertion 程式碼、expected value 或驗證邏輯。
  • 不允許 AI 自動 merge PR。
  • 不將 network/server busy、環境問題、產品缺陷誤列為一般 healing pass。
  • P1 不處理完整自動產生新 test case；此項列為 P2 延伸。

五、整體架構與流程圖

整體架構分成四層：原始測試執行、失敗證據收集、AI 分析與修復、人工審核與知識沉澱。這樣可以保留原始測試結果，又能讓 AI 在不破壞品質訊號的前提下協助維護測試。

因目前只有一台 iOS 測試裝置，所有需要實際操作 App 的 replay、healing verification、dependency recovery 都必須排程。系統採用 device-aware queue，不把所有 failure 都同步 healing。

六、Failure 類型與 AI 處理邊界

Failure 類型AI 可做到什麼結果標註
Locator / Page Object Drift產生 locator 或 page object patch，replay 通過後自動開 PRpass_with_healing
Readiness / Wait Drift改成狀態型等待，例如 visible、enabled、spinner gone、API completepass_with_healing
Micro-flow Drift新增語意明確的中介步驟，例如 Continue / Open / Next，但不改 assertionpass_with_healing
Assertion Failure Symptom追 root cause；若原因是 locator、wait、flow、network 等，仍可 healing依 root cause 決定
Assertion Logic Change Required不修改 assertion，分流到 new test case / redesignneeds_new_test_case
Dependency Cascade找出 upstream root failure，downstream case 不算獨立 failblocked_by_upstream
Network / Server Busy依 retry policy 重跑與統計，不修改測試碼network_issue / server_busy
Product Bug Suspected收集證據、標出可能 root cause，不做 healingproduct_bug_suspected

七、執行流程

  1. 每個 case 執行時即時拉取 evidence。
  2. Pass case 保留 compact baseline evidence，刪除非必要 artifact。
  3. Fail case 保留完整 screenshot、log、accessibility tree、network evidence、trace。
  4. 先做非 AI 初步 fail reason 分析。
  5. 再交給 AI 做正式 root cause analysis。
  6. 判斷是否需要立即 retry / recovery / repair。
  7. 非緊急 case 等整份 AT 跑完後進入 deferred healing queue。
  8. Replay 通過後產出 HTML 報告並自動開 PR。

八、Device-aware 排程

類型排程
Network / server busy可 immediate retry
Setup / dependency root failure可 immediate recovery
Smoke / blocking case可 immediate retry / recovery
Locator / page object driftdeferred healing queue
Micro-flow drift預設 deferred，若 blocking 才 immediate
Product bug suspected不進 healing

九、HTML Report Review Gate

最後會產出一份 HTML 報告，作為人工審核 AI healing PR 的主要入口。Reviewer 不需要先逐行查 raw log，也能看到 failure 原因、截圖、修復方式、replay 證據與風險標記。

報告區塊內容
Run Summarybranch、commit、environment、device、App version、test run id
Failure Detailcase 名稱、suite、owner、original status、final status
Screenshot Evidencefailure screenshot、healed replay screenshot、before/after 對照
Error Reason非 AI 初步 reason、AI root cause reason、confidence
Action Takenimmediate retry、deferred healing、data recovery、dependency recovery 等
Code Change Summary修改檔案、修改類型、diff 摘要
Replay Evidencereplay command、結果、執行時間、裝置與環境
Risk Flags是否碰到 assertion、是否 major flow、是否 flaky、是否 blocking
若報告顯示 assertion logic change、major flow change、product bug suspected 或 manual_review_required，不能只靠報告 approve merge，必須轉人工深入 review。

十一、主要風險與控管

風險控管方式
AI 把真正 product bug 修成 pass禁止修改 assertion；保留 original failure；product bug suspected 不進 healing
只有一台 iOS device 導致時間失控採 device-aware queue；只有 high-priority failure immediate retry/recovery
Flaky 被誤判成修復成功區分 pass_after_retry 與 pass_with_healing
Report 摘要不足，reviewer 看不出風險HTML report 必含截圖、root cause、diff 摘要、replay 證據與 risk flags
AI 修改範圍過大只允許低風險測試路徑 drift；major flow 與 assertion logic change 轉人工

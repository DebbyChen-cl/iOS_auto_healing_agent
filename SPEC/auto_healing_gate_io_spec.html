AI Auto-Healing Gate I/O 實作規格

本文件定義 PHD iOS App UI 自動化測試導入 AI Auto-Healing 時，每一個 gate 的輸入、輸出、處理責任與通過條件。此文件用於實際實作拆工與介面對齊，不使用 JSON 描述。

實作規格 iOS UI Testing L3 Auto-Healing 不自動 Merge 2026-06-24

一、Gate 設計原則

系統的主線不是讓 AI 把測試變綠，而是讓每一次失敗都能被收集、分類、分析、修復、驗證、審核，最後保留成可追溯的品質訊號。
原則實作含義
原始結果不可被覆蓋原始 test runner 的 pass/fail 永遠保存；AI 修復後只能產生新的 healed result。
Assertion 不可被 AI 修改AI 可以修造成 assertion failure 的 root cause，但不能改 assertion 程式碼、expected value 或驗證邏輯。
需要操作 App 的工作要排程目前只有一台 iOS device，replay、healing verification、dependency recovery 都要走 device-aware queue。
修復成功不等於一般 passAI 修復後通過要標成 pass_with_healing，不可標成 pass。
Review 後才沉澱知識Auto-healing event 不能直接進長期 knowledge store；要等人工 review 結果後再萃取。

四、各 Gate 詳細規格

Test Identity Gate
G01

在測試開始前，先準備每個 test case 的身份資料。這是後續分類、報告、PR、knowledge promotion 的索引基礎。詳細欄位定義請參考 test_identity_spec.md。

Input
  • 測試清單與 case 名稱
  • 測試檔案路徑與 function / method 名稱
  • 對應 feature、user flow、priority、owner
  • 是否屬於 smoke、blocking、non-blocking case
  • 已知 upstream / downstream dependency
Output
  • 唯一 test identity
  • feature 與 user flow 對應關係
  • priority 與 blocking 屬性
  • dependency relationship 初始資料
通過條件每個 case 都能對應到穩定 identity；至少包含 case 名稱、feature、priority。
失敗去向缺 metadata 的 case 仍可執行，但標為 identity_incomplete，後續報告提示補齊。
Context Capture Gate
G02

每個 case 執行時即時收集環境與裝置上下文。這些資料用來判斷失敗是否與 App version、device、iOS version 或環境有關。

Input
  • test identity
  • branch、commit、build number、App version
  • iOS version、device model、device id
  • 測試環境、登入帳號或 fixture profile
  • case 開始時間與 runner 狀態
Output
  • runtime context
  • case execution session id
  • 當下 device / App 狀態摘要
  • 後續 evidence 的關聯索引
通過條件每次 case execution 都能留下可追溯的 context。
失敗去向若 context capture 不完整，仍繼續跑測試，但在 report 標出 evidence gap。
Previous Result Lookup Gate
G03

在本次 case 結果進入 pass/fail 分流前，自動查詢上一輪同一 case 的狀態。這是 flaky 判斷、regression 判斷、healing 優先級與 HTML report 趨勢說明的重要輸入。詳細規則請參考 previous_result_history_spec.md。

Input
  • test identity
  • 目前 branch、commit、build number
  • 目標比較基準，例如上一輪同 branch、上一個 main/nightly run、上一個 release baseline
  • 歷史 test result store
  • 歷史 HTML report 或 CI run record
Output
  • 上一輪 case 狀態，例如 pass、fail、pass_with_healing、flaky_suspect
  • 上一輪 failure type 與 final status
  • 上一輪是否有 healing PR 與 reviewer decision
  • 最近 N 次狀態摘要
  • 是否從 pass 轉 fail、fail 轉 pass、持續 fail、反覆不穩定
自動取得方式優先從本系統的 Result Store 查詢；若尚未建立，先從 CI artifact、歷史 HTML report、test management export 或 GitHub Actions/Jenkins run metadata 匯入。查詢 key 應以 stable test identity 為主，不只靠 case name。
通過條件能找到上一輪狀態，或明確標記為 no_previous_result。
失敗去向查不到歷史資料時不阻塞測試，輸出 no_previous_result，並在 report 標示 history unavailable。
Pass Baseline Gate
G04

Pass case 不保存完整重型 artifact，只保留未來 AI 比對正常狀態所需的 step-level baseline。詳細保存範圍與補抓規則請參考 pass_baseline_retention_spec.md。

Input
  • 原始 pass result
  • runtime context
  • 上一輪 case 狀態
  • 關鍵 screen state
  • 必要 screenshot 或 screenshot hash
  • accessibility tree 摘要
Output
  • compact baseline evidence
  • pass status
  • 可供未來失敗比對的正常 UI 狀態
  • 非必要 artifact 刪除紀錄
通過條件保留足以做未來比對的最小正常狀態，不保存過量 video / trace。
失敗去向若 baseline 保存失敗，不影響 case pass，但記錄為 baseline collection warning。
Failure Evidence Gate
G05

Fail case 必須保留足夠 evidence。這是 AI 分析 root cause 與 HTML report 的核心素材。必收與條件收資料請參考 failure_evidence_collection_spec.md。

Input
  • 原始 fail result
  • 上一輪 case 狀態
  • error message、stack trace、assertion message
  • failure screenshot、必要時 video / trace
  • accessibility tree、visible element list
  • network request / response evidence
  • 當下 screen name、App state、fixture state
Output
  • full failure evidence package
  • original failure snapshot
  • error signature
  • AI analysis 的輸入資料
通過條件至少有 error、screenshot、accessibility tree 或可替代的 UI state evidence。
失敗去向若 evidence 不足，進入 manual_review_required，不允許 L3 auto-healing。
Preliminary Classifier Gate
G06

先用非 AI 規則做初步分類，減少 AI 誤判，也讓 network/server busy、device crash 等高頻類型快速分流。分類優先順序請參考 failure_taxonomy_priority_spec.md。

Input
  • full failure evidence package
  • 上一輪 case 狀態與最近 N 次狀態摘要
  • runner error code
  • timeout、network status、HTTP status
  • device / App crash signal
  • assertion symptom 與 error signature
Output
  • 初步 failure category
  • 是否疑似 network/server busy
  • 是否疑似 infra / device issue
  • 是否需要 AI deeper analysis
通過條件產生至少一個初步分類，並指出分類信心與需要 AI 判斷的部分。
失敗去向分類不足時仍交給 AI，但在 report 標示 preliminary classifier inconclusive。
AI Root Cause Gate
G07

AI 根據 failure evidence、初步分類、歷史 baseline 與 App 使用知識，分析真正 root cause。Assertion failure 在此只視為症狀，不直接視為 root cause。分類優先順序請參考 failure_taxonomy_priority_spec.md。

Input
  • failure evidence package
  • preliminary category
  • 上一輪 case 狀態與歷史趨勢
  • pass baseline 對照資料
  • App usage knowledge
  • test fragility knowledge
  • 歷史相似 failure / healing records
Output
  • root cause reason
  • failure type
  • confidence
  • suggested action
  • 是否可進 L3 healing
  • risk flags
通過條件AI 能說明失敗原因、對應 failure taxonomy，並提出 action 或 no-healing 理由。
失敗去向低信心、缺 evidence、疑似產品問題或需要改 assertion 時，輸出 manual_review_required 或 needs_new_test_case。
Scheduling Decision Gate
G08

決定這次 failure 是否需要立即處理，或等 full AT run 結束後再進 deferred healing queue。device 排程與 lane 規則請參考 immediate_vs_deferred_policy_spec.md，L3 與 immediate healing eligibility 請參考 l3_healing_eligibility_spec.md。

Input
  • root cause analysis
  • 上一輪 case 狀態與是否 regression
  • case priority 與 blocking 屬性
  • device availability
  • retry / recovery budget
  • 目前 AT run 進度
Output
  • immediate action
  • deferred healing
  • no healing
  • manual review
  • 排程理由
通過條件每個 fail case 都有明確 lane：immediate、deferred、manual、no-healing。
失敗去向若排程無法決定，預設不搶 device，進 deferred 或 manual review。
Immediate Action Gate
G09

只處理高優先 failure，例如 app crash rerun、network/server busy immediate retry、generation fail retry、setup / dependency root failure recovery、smoke / blocking case。v1 所有 immediate rerun 最多一次，device 排程請參考 immediate_vs_deferred_policy_spec.md，retry 標註與後續流程請參考 retry_policy_spec.md。

Input
  • immediate action decision
  • 可用 device
  • retry / recovery policy
  • 原始 failure evidence
  • case identity 與 dependency info
Output
  • retry result
  • recovery result
  • pass_after_retry 或 pass_after_recovery 類狀態
  • 仍失敗時的 updated evidence
通過條件在 budget 內完成 retry / recovery，且不污染原始 failure record。
失敗去向超過 budget 或仍失敗，進 Final Reconciliation，不再無限制重跑。
Deferred Healing Queue Gate
G10

非緊急但可 healing 的 failure 先排隊，等 full AT run 完成後再依優先級使用 device 進行修復與 replay。

Input
  • healing candidate list
  • full AT run result
  • root cause grouping
  • dependency cascade result
  • device queue policy
Output
  • 排序後的 healing tasks
  • 每個 task 的 priority
  • 被略過的 no-healing tasks
  • queue execution plan
通過條件queue 只包含符合 L3 或 L2 review 條件的 healing candidate。
失敗去向若 candidate 被 root cause grouping 判定為 downstream blocked，改標 blocked_by_upstream，不進 healing。
Patch Generation Gate
G11

AI 產生測試碼修補，但只能在允許範圍內修改 locator、page object、狀態型等待或 micro-flow step。可進 L3 的修補範圍請參考 l3_healing_eligibility_spec.md。若 locator 位於 shared page object，必須依 replay_verification_spec.md 的 Shared Locator Impact Gate 決定是否 global update 或改用 locator override parameter。

Input
  • root cause reason
  • full failure evidence
  • 相關測試檔與 page object
  • 允許修改範圍
  • 禁止修改規則，例如 assertion 不可改
Output
  • candidate patch
  • changed files list
  • diff summary
  • healing action description
  • risk flags
通過條件patch 不改 assertion，不改 expected value，不繞過失敗，且修改範圍局部。
失敗去向若需要大改流程或 assertion，停止 patch，標 needs_new_test_case 或 manual_review_required。
Replay Verification Gate
G12

修復是否有效只能由 replay 驗證，不接受 AI 自稱修好。Replay 最多允許 3 個 patch attempts；同一份 patch 不重複碰運氣。Replay scope、shared locator fallback 與通過條件請以 replay_verification_spec.md 為準。

Input
  • candidate patch
  • 原始 fail case
  • replay command
  • device 與環境設定
  • replay budget 與重跑次數限制
  • shared locator impact 判斷
Output
  • replay pass / fail
  • replay screenshot
  • replay log
  • replay attempt count
  • final status candidate
  • healing succeeded / failed
通過條件套用 patch 後 replay 通過、原 fail step 通過、final assertion 通過、沒有新增高風險 signal；若使用 locator override parameter，default path 不可被破壞。
失敗去向Replay 不通過時，若仍屬低風險且 root cause 明確，可產生下一版 patch 後再 replay，總計最多 3 attempts；超過上限或需改 assertion / expected value 時，停止 L3 healing。
Final Reconciliation Gate
G13

將原始測試結果、AI 分析、replay、dependency graph 合併成最終報告狀態。這一層負責保護品質訊號。

Input
  • original test result
  • 上一輪 case 狀態與最近 N 次趨勢
  • failure taxonomy result
  • dependency graph
  • retry / recovery result
  • healing replay result
Output
  • final status
  • 狀態變化說明，例如 new failure、recovered、repeated failure、flaky trend
  • root failure group
  • blocked_by_upstream mapping
  • report-ready summary
  • PR eligibility
通過條件每個 case 只有一個 final status，但保留 original status 與 healing/recovery status。
失敗去向無法 reconcile 時標 manual_review_required，不可自動 merge。
HTML Report Gate
G14

產出 reviewer 主要審核介面。報告必須讓 reviewer 看見失敗截圖、原因、AI 修了什麼、replay 是否通過，以及是否有風險。L3 healing 的 HTML report 採嚴格 approval rule；只有符合 html_report_approval_rule_spec.md 的 report 才能作為 merge gate。Run-level 與 trend-level metrics 請參考 metrics_spec.md。

Input
  • run summary
  • 上一輪 case 狀態與狀態變化說明
  • failure detail
  • failure screenshot 與 replay screenshot
  • root cause reason
  • patch diff summary
  • replay evidence
  • risk flags
  • PR link
  • approval checklist
  • metrics summary
Output
  • HTML review report
  • reviewer decision area
  • case-level status table
  • PR merge readiness summary
  • approval decision candidate
  • run-level metrics summary
通過條件嚴格 checklist 全部通過，且 reviewer 可在不先看 raw log 的情況下判斷 AI healing 可信、patch 合法、replay 有效；report 必須獨立統計 pass_with_healing、retry、infra / service issue 與品質 guardrail。
失敗去向缺截圖、缺 diff summary、缺 replay evidence、root cause 不明、patch 超出 L3 邊界或 PR diff 不一致時，報告不可作為 merge gate。
PR Review Gate
G15

AI healing 成功後可自動開 run-level PR，但 merge 必須由人工根據 HTML report 審核。每輪 AT run 最多一個 Auto-Healing PR；PR 只包含 eligible healing patches，不合格 case 只出現在 report 中。PR generation 詳細規則請參考 pr_generation_rule_spec.md。只有 HTML report decision 為 approve_merge_allowed 時，PR 才能進入 merge allowed 狀態。

Input
  • run-level HTML report
  • run-level AI healing PR
  • included healing cases
  • excluded cases 與排除原因
  • changed files
  • diff summary
  • risk flags
  • replay result
  • approval checklist result
Output
  • approve_merge_allowed
  • reject_healing
  • need_more_evidence
  • manual_code_review_required
  • review comments
  • merge allowed / not allowed
  • PR merge gate status
通過條件只有 approve_merge_allowed 才允許 merge；approve 代表接受 run-level PR 內所有 included pass_with_healing patch 的證據，不代表原始測試一般 pass。
失敗去向reject_healing、need_more_evidence、manual_code_review_required 都不可 merge；不合格 case 不可放進 PR，但必須留在 HTML report 中說明。
Knowledge Promotion Gate
G16

只有通過 review 的事件，才可萃取成長期知識。Knowledge 分為三類：App Usage Knowledge、Test Fragility Knowledge、Coverage Opportunity Knowledge。Promotion eligibility、confidence 與不可 promotion 的內容請參考 knowledge_promotion_spec.md；若 knowledge 要提供給 New Case Generator，讀取邊界請參考 new_case_generator_boundary_spec.md。

Input
  • approve_merge_allowed auto-healing event
  • root cause analysis
  • healing action
  • replay result
  • reviewer decision 與 comments
  • 是否反覆出現的 failure pattern
  • source evidence summary
Output
  • App Usage Knowledge
  • Test Fragility Knowledge
  • Coverage Opportunity Knowledge
  • source event reference
  • confidence：candidate / trusted / deprecated
  • 不可 promotion 的 event history record
通過條件只有 approve_merge_allowed 且 source evidence 完整的 event 才能 promotion；單次 approved event 先進 candidate，重複 approved 或 reviewer 明確確認後才能成為 trusted。
失敗去向reject_healing、need_more_evidence、manual_code_review_required、單次 flaky、network/server busy、generation fail 只能保留在 event history 或 metrics / trend，不可提升為 auto-healing knowledge。

五、狀態輸出規則

Final Status產生條件可否作為一般 Pass
pass原始測試直接通過，未經 retry、healing、recovery。可以
pass_with_healingAI 修改測試路徑、locator、page object 或 micro-flow 後，replay 通過。不可以，需 report review
pass_after_retry不改 code 重跑後通過。不可以，代表疑似 flaky
pass_after_network_retrynetwork/server busy 依 retry policy 重跑後通過。不可以，需獨立統計
blocked_by_upstream此 case 因 upstream failure 無法成立。不可以，不算獨立 fail
needs_new_test_case需要修改測試意圖、assertion 邏輯或主流程 redesign。不可以
product_bug_suspected失敗可能代表產品缺陷。不可以
manual_review_requiredAI 無法安全判斷或證據不足。不可以

六、實作拆工建議

階段先做的 Gate完成標準
P0-1G01 到 G06能收集 pass baseline、fail evidence、上一輪 case 狀態，並做非 AI 初步分類。
P0-2G07、G08、G13、G14AI 可分析 root cause，產出含歷史狀態對照的 HTML report，但先不修 code。
P1-1G10 到 G12支援 deferred healing queue、patch generation、replay verification。
P1-2G15AI healing PR 與 HTML report review gate 串起來，不自動 merge。
P2G16從 approved event 萃取三類 knowledge，供 future new case generator 使用。
